Node.js `net` 模块实战使用频率、知识点与落地场景

## 一、先给结论

 **普通业务开发（CRUD、Web 接口、前后端服务）：** **`net`** ** 极少使用，属于低频 API** 。
 **底层服务、网络中间件、长连接、TCP 通信、内网服务、网关类项目：** **`net`** ** 是高频刚需，属于高配技能** 。

简单划分：

1. 90% 日常业务（HTTP 接口、静态服务、管理后台、小程序 / APP 接口）：只用 `http`/`express`/`koa`， **基本碰不到 net** ；
2. 网络编程、通信服务、物联网、游戏服务、代理网关、TCP 长连接、端口转发、自定义协议服务： **net 核心主力** 。

---

## 二、基础核心知识点（必掌握）

`net` 是 Node.js  **原生 TCP 套接字模块** ，工作在 **传输层** ，只负责二进制数据流收发， **不内置任何应用层协议** （HTTP、WebSocket、MQTT 等都要自己解析 / 拼装）。

### 1. 核心两类角色

#### （1）TCP 服务端 `net.createServer()`

用于监听端口，接收客户端 TCP 连接，对应你上一段代码。
核心事件：

* `listening`：服务成功监听端口触发
* `connection`： **有客户端建立 TCP 连接** （每个连接对应一个独立 `socket`）
* `close`：服务关闭
* `error`：服务异常

#### （2）TCP 客户端 `net.connect()` / `net.createConnection()`

主动向远端 TCP 服务发起连接，用于 **请求其他 TCP 服务** 。

#### （3）核心对象：`socket` 套接字

连接建立后，两端都通过 `socket` 通信，本质是 **双向数据流** ：

* `data`：收到对方发来的二进制数据（`Buffer`）
* `end`：对方主动关闭连接
* `close`：连接彻底断开
* `error`：读写异常、断网、超时
* `write()`：向对方发送数据
* `end()`：发送数据并关闭当前连接
* `setTimeout(ms)`：设置连接超时（空闲超时自动断开）

### 2. 关键特性

1. **面向字节流** ：TCP 是流式协议，数据无严格 “消息边界”，**粘包 / 拆包**是最经典的坑（实战重点）；
2. **默认长连接** ：TCP 连接建立后，不会自动断开，可反复收发数据；
3. 纯二进制传输：收发都是 `Buffer`，字符串需手动转码；
4. 无协议约束：你可以自定义私有通信协议。

### 3. 和 `http`/`WebSocket` 关系

* `http`： **基于 net 封装** ，自动解析 HTTP 报文、状态码、请求头，屏蔽 TCP 底层；
* `WebSocket`：握手阶段基于 HTTP，数据传输阶段本质也是 TCP；
* 学习顺序：`net` → 理解 TCP → 再理解 HTTP、WebSocket 底层原理。

## 三、实战高频应用场景（项目里真正常用的场景）

### 场景 1：内网 TCP 服务 / 跨语言通信（最常用）

企业内部多语言混合开发很常见：Java/Python/Go/C++ 服务 <-> Node.js 服务，很多会采用 **TCP 私有协议** 通信。

* 例子：硬件采集服务、后台计算服务、本地工具服务，对外暴露 TCP 端口，Node 作为网关 / 中间层对接。
* 用法：Node 用 `net` 做  **TCP 客户端** ，连接远端服务，转发数据。

**极简 TCP 客户端示例**

```javascript
const net = require('net');

// 连接远端 TCP 服务 IP:端口
const client = net.connect({ host: '127.0.0.1', port: 8888 }, () => {
  console.log('已连接到 TCP 服务端');
  // 发送数据
  client.write('Hello TCP Server');
});

// 接收服务端返回数据
client.on('data', (chunk) => {
  console.log('收到数据：', chunk.toString('utf8'));
});

// 连接关闭
client.on('end', () => {
  console.log('连接断开');
});

// 异常捕获
client.on('error', (err) => {
  console.error('连接异常：', err);
});
```

---

### 场景 2：物联网 / 硬件设备通信（工业、IoT 项目主流）

绝大多数嵌入式设备、传感器、PLC、串口转发设备，都使用 **TCP 长连接** 上报数据。

* 设备主动连 Node TCP 服务，定时上报温度、状态、告警；
* Node 下发指令控制设备；
* 特点： **大量长连接、高并发、数据高频上报** 。

---

### 场景 3：端口转发 / 简单代理服务（运维、网关工具）

用 `net` 实现 **端口映射、流量转发** ，常见于内网穿透、本地代理、测试环境转发。
原理：客户端连 Node TCP 服务 → Node 再转发到目标服务，双向透传数据。

简易端口转发核心逻辑：

```javascript
const net = require('net');

// 本地监听 9999，转发到 127.0.0.1:8080
net.createServer((clientSocket) => {
  // 连接目标服务
  const remoteSocket = net.connect(8080, '127.0.0.1');
  // 双向数据转发
  clientSocket.pipe(remoteSocket);
  remoteSocket.pipe(clientSocket);
}).listen(9999, () => {
  console.log('端口转发启动，监听 9999');
});
```

---

### 场景 4：游戏服务端 / 联机小游戏

休闲小游戏、联机互动程序，追求 **低延迟** ，不会用 HTTP（短连接、头信息冗余），普遍用 TCP 长连接。

* 玩家登录、位置同步、动作指令、房间通信；
* Node + `net` 做轻量游戏网关 / 房间服务。

---

### 场景 5：自定义私有协议服务

公司内部自研通信协议、文件传输协议、消息推送服务，不想依赖 HTTP 臃肿头部，基于 `net` 自定义格式。
常见格式：`消息长度 + 消息体`（解决 TCP 粘包问题）。

---

### 场景 6：模拟网络服务 / 接口 Mock（开发测试）

本地模拟第三方 TCP 服务，用于联调测试，不用部署完整第三方环境。

---

### 场景 7：日志收集、数据上报中转

部分传统日志采集工具基于 TCP 上报，Node 用 `net` 接收日志，再转存文件 / 数据库。

## 四、实战必踩坑 & 解决方案（重中之重）

### 1. 最大坑：TCP 粘包 / 拆包

#### 现象

TCP 是 **流协议** ，不保证每次 `data` 事件对应 “一条完整消息”：

* 多条消息粘在一起（粘包）；
* 一条消息被分成多次接收（拆包）。

#### 行业通用解决方案（生产环境标准做法）

**定长报文头 + 消息体长度**
通信格式约定：
`[4字节长度] + [消息体数据]`
流程：

1. 发送方：先把消息体长度转为 4 字节二进制，再拼接消息体一起发送；
2. 接收方：先读 4 字节拿到总长度，再根据长度截取完整消息。

> 只要做正式 TCP 通信， **必须处理消息边界** ，否则业务直接错乱。

### 2. 连接泄漏（长连接项目高频问题）

* 客户端异常断网、闪退，服务端 `socket` 不会立刻触发 `end`；
* 大量僵尸连接占用端口和内存。

 **解决** ：给所有 `socket` 设置**空闲超时**

```javascript
socket.setTimeout(30000); // 30秒无数据自动超时
socket.on('timeout', () => {
  console.log('连接空闲超时，主动断开');
  socket.end();
});
```

### 3. 异常监听不全导致服务崩溃

`socket`、server 所有 `error` 事件 **必须监听** ，Node 事件未捕获异常会直接进程退出。

### 4. 编码问题

默认传输 `Buffer`，字符串收发必须统一编码（推荐 `utf8`），两端编码不一致会乱码。

### 5. 并发连接上限

`net` 单机可承载上万长连接，但要配合系统文件句柄调优，高并发 IoT / 游戏场景需要运维配合。

## 五、`net` 与相近技术选型对比（实战怎么选）

| 技术              | 层级               | 适用场景                                     | 业务推荐度                 |
| ----------------- | ------------------ | -------------------------------------------- | -------------------------- |
| `net`(TCP)      | 传输层             | 低延迟长连接、硬件、游戏、私有协议、内网通信 | 底层 / 硬件 / 游戏项目首选 |
| `http`          | 应用层（基于 TCP） | 标准 Web 接口、前后端交互                    | 绝大多数互联网业务首选     |
| `ws`(WebSocket) | 应用层（基于 TCP） | 网页长连接、在线聊天、实时推送               | 浏览器端长连接首选         |

**总结选型口诀**

1. 做网页 / APP 普通接口 → 用 `http/express/koa`；
2. 浏览器需要实时长连接 → 用 `WebSocket`；
3. 硬件、游戏、多语言内网通信、自定义协议、端口转发 → 用 `net`；
4. 单纯传文件、简单通信，优先成熟协议， **不要盲目手写 TCP 私有协议** 。

## 六、学习建议（分阶段）

### 1. 入门阶段（必会）

1. 掌握 `net` 服务端 + 客户端基础写法；
2. 理解 `socket` 核心事件：`data` / `end` / `close` / `error`；
3. 会基础二进制 `Buffer` 读写、转字符串。

### 2. 进阶（项目上岗必备）

1. 吃透 **TCP 粘包拆包** 解决方案（定长包头方案）；
2. 连接超时、心跳包机制（长连接保活）；
3. 连接池管理、异常重连。

### 3. 高阶（架构 / 底层开发）

1. 高并发连接调优、事件队列、内存监控；
2. 基于 `net` 封装自定义协议框架。

---

## 七、最终总结

1. **普通 CRUD 业务** ：`net` 属于 **冷门** ，日常基本不用，了解原理即可；
2. **IoT、硬件、游戏、内网多语言通信、网关代理、自定义协议** ：`net` 是 **高频核心技术** ，属于面试 + 实战高配技能；
3. 学习价值：学会 `net` 能彻底搞懂 TCP 网络模型，反过来加深对 `http`、`WebSocket` 的理解，是 Node 网络编程的分水岭。
