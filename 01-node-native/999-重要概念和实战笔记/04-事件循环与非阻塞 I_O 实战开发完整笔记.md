# Node\.js 事件循环与非阻塞 I/O 实战开发完整笔记

## 一、核心概念（工作开发真正的理解）

### 1\. 什么是 非阻塞 I/O？

Node\.js 所有性能、高并发的根基就是：**单线程 \+ 非阻塞 I/O \+ 事件循环**。

**I/O 场景**：数据库查询、网络请求、文件读写、加密压缩、TCP 通信（都是耗时操作）。

- **阻塞 I/O**：发起耗时操作 → 主线程原地死等 → 做完才能走下一行 → 服务无法处理新请求。

- **非阻塞 I/O**：发起耗时操作 → **主线程不等待，立刻继续执行代码** → 耗时任务交给内核 / libuv 线程池后台处理 → 完成后把回调放入队列 → 主线程空闲再执行回调。

**一句话总结**：I/O 在后台慢慢跑，主线程全力处理新请求，不浪费一秒 CPU。

### 2\. 什么是 事件循环 Event Loop？

事件循环是 Node 主线程的**异步回调调度机制**，由底层 libuv 实现。

作用：让单线程可以有序、轮流执行所有异步回调，不会乱序、不会卡死。

**主线程完整执行流程**：

1. 先跑完所有 **同步代码**

2. 一次性清空所有 **微任务队列**

3. 按固定阶段轮流执行 **宏任务队列**

4. 队列为空则挂起等待新事件，循环往复

### 3\. 底层依赖 libuv（面试\+排查问题必懂）

- Node 事件循环、线程池、I/O 多路复用全部由 libuv 实现

- 默认开启 **4 条线程池**，专门处理：文件 I/O、crypto 加密、zlib 压缩等阻塞任务

- JS 主线程只做调度，不做耗时阻塞任务

## 二、事件循环严格执行优先级（工作排错核心）

### 完整执行顺序（必背）

**同步代码 \> 微任务 \> 宏任务（分阶段）**

#### 1\. 同步代码（最先执行）

#### 2\. 微任务队列（一次性全部清空）

- `process.nextTick` （优先级最高，内部微任务）

- Promise\.then / catch / finally

#### 3\. 宏任务队列（一轮循环只走一个阶段）

1. **timers**：setTimeout / setInterval

2. pending callbacks：延迟 I/O 回调

3. idle/prepare：内部预留（业务无感知）

4. **poll**：网络、文件 I/O 回调（业务最多）

5. **check**：setImmediate

6. close callbacks：socket / stream 关闭事件

### 优先级验证代码（可直接运行）

```javascript
console.log('1. 同步代码')

setTimeout(() => console.log('5. setTimeout 宏任务'), 0)
setImmediate(() => console.log('6. setImmediate 宏任务'))

Promise.resolve().then(() => {
  console.log('4. Promise 微任务')
})

process.nextTick(() => {
  console.log('3. nextTick 最高微任务')
})

console.log('2. 同步代码末尾')

```

**固定输出顺序**：同步代码 → nextTick → Promise → setTimeout → setImmediate

## 三、企业实战工作场景（哪些业务依赖事件循环）

**所有 Node/Express 后端服务，本质都是事件循环 \+ 非阻塞 I/O 跑起来的**

### 场景1：Express/HTTP 高并发接口（核心场景 ⭐⭐⭐⭐⭐）

每一个用户请求、数据库查询、Redis 查询、第三方接口调用，都是**非阻塞异步 I/O**。

I/O 等待过程中，主线程不会卡住，可以继续接收成千上万个新请求，这就是 Node 高并发的本质。

```javascript
app.get('/list', async (req, res) => {
  // 数据库查询：非阻塞，不占用主线程
  const data = await db.query('select * from user')
  res.json(data)
})

```

### 场景2：大文件流式上传/下载（fs stream）⭐⭐⭐⭐⭐

`createReadStream / createWriteStream` 基于事件循环 poll 阶段回调，非阻塞读写大文件，不会炸内存、不会卡死服务。

### 场景3：定时任务、延时重试 ⭐⭐⭐⭐

setTimeout / setInterval 由 timers 阶段调度，用于：缓存清理、日志切割、失败重试、定时统计。

### 场景4：Socket/TCP 长连接、设备上报 ⭐⭐⭐⭐

大量物联网设备、聊天长连接，数据事件由 poll 阶段持续调度，支持上万并发连接。

### 场景5：Crypto 加密、zlib 压缩 ⭐⭐⭐

属于 CPU 密集任务，交给 libuv 线程池异步处理，**不阻塞事件循环**。

### 场景6：消息队列消费、异步任务轮询 ⭐⭐⭐

RabbitMQ、MQTT 等待消息属于网络 I/O，全程非阻塞，不影响接口服务。

## 四、工作开发重点注意事项（线上90%故障来源）

### 1\. **主线程禁止任何 CPU 密集同步计算（最致命）**

超大 for 循环、超大 JSON 解析、复杂正则、大数运算、同步递归，会直接**阻塞事件循环**，导致所有接口超时、服务假死。

❌ 错误高危写法

```javascript
// 绝对禁止在接口主线程执行重度计算
function heavyCalc() {
  let arr = new Array(10000000).fill(1)
  let sum = 0
  for (let i of arr) sum += i
}
heavyCalc() 

```

✅ 解决方案

- 大循环分片执行，释放事件循环

- 重度计算丢入 `child_process` 子进程

- 独立计算服务，HTTP 远程调用解耦

### 2\. **禁止使用同步 I/O 方法**

`fs.readFileSync / readdirSync` 等同步文件操作，会直接阻塞主线程，高并发项目**全线禁用**。

✅ 统一使用 `fs.promises` 异步 API

### 3\. 禁止微任务无限递归（卡死整站）

nextTick / Promise 无限递归，会无限塞满微任务队列，**宏任务、接口、定时器全部不执行**，服务彻底卡死。

❌ 高危代码

```javascript
function loop() {
  process.nextTick(loop)
}
loop()

```

### 4\. libuv 线程池耗尽导致服务卡顿

默认线程池只有 4 条，大量文件操作、加密、压缩会占满线程池，新任务排队阻塞。

✅ 项目入口顶部全局优化（线上常用）

```javascript
// 放大线程池上限，提升异步处理能力
process.env.UV_THREADPOOL_SIZE = 16

```

### 5\. 不要依赖 setTimeout 精准定时

setTimeout 0 不是立即执行，必须等本轮同步、微任务结束，且进入下一轮 timers 阶段才执行，存在延迟误差。高精度定时必须使用专业定时库。

### 6\. 回调内部同步代码不要过长

单个异步回调内部写大量同步逻辑，会拉长单次事件循环耗时，整体接口响应变慢、并发下降。

### 7\. 明确区分：哪些操作阻塞 / 不阻塞

**✅ 不阻塞事件循环（安全）**

- 异步文件 I/O、网络请求、数据库操作

- crypto、zlib 线程池任务

- Promise、async/await、定时器回调

**❌ 阻塞事件循环（高危）**

- 超大循环、复杂同步计算、死循环

- 同步文件 I/O

- 超大 JSON\.parse、复杂正则匹配

## 五、企业线上开发规范（落地标准）

1. 所有 I/O 统一使用异步 API，杜绝一切同步阻塞 I/O

2. 主线程只处理轻量业务逻辑，重度 CPU 计算强制解耦到子进程/独立服务

3. 严禁无限递归微任务，防止事件循环卡死

4. 高 IO、加密场景，主动调大 libuv 线程池

5. 监控事件循环延迟，超过阈值告警，快速定位阻塞代码

6. 定时器仅做低精度轮询，不做核心精准业务

## 六、面试高频简答（工作\+面试通用）

#### 1\. Node 单线程为什么能支持高并发？

基于**非阻塞 I/O \+ 事件循环**，I/O 等待耗时不占用主线程，主线程持续处理新请求，无多线程切换开销，并发性能极强。

#### 2\. 事件循环执行顺序？

同步代码 → nextTick → Promise微任务 → 宏任务（timers \-\> poll \-\> check）

#### 3\. 什么操作会阻塞整个 Node 服务？

主线程 CPU 密集计算、同步文件 I/O、死循环、无限递归微任务。

#### 4\. setTimeout 和 setImmediate 谁先执行？

全局顶层调用顺序不固定；**在 I/O 回调内部调用，setImmediate 永远优先执行**。

#### 5\. libuv 线程池作用？

处理文件 I/O、加密、压缩等阻塞型任务，将耗时操作脱离主线程，保证事件循环不卡顿。
